I have been thinking a lot about how to create general intelligence since 2008, but due to life-circumstances, I have not had the time to properly explore these ideas. The following is my current approach to taking baby-steps toward AGI.

Outside the Box

Traditional neural networks are nearing their limits. LLMs are constrained by training data, while other machine learning algorithms have narrow applications. Leading AI researchers acknowledge this, noting that current approaches largely build on decades-old concepts rather than introducing novel ideas.

AGI might be achievable through a combination of virtual machine-executed programs linked to small neural networks, along with efficient input signal identification from streams or static length inputs. These signals trigger specific program sets within the VM. Program generation would be methodical yet intelligent, utilizing a form of smart brute-force approach. This intelligent brute-force method employs reductionism to learn new problems through input/output examples. Problems with only input (only X, no y) are translated into smaller input-output recognition tasks for both signature and sequence training. Background processes (a subconcious?) then reorganize signatures and programs, identifying sets that solve multiple problems or work across various signals. This system would optimize or combine existing programs to address multiple challenges simultaneously.

Brute-force Reductionism

In this example, the pattern is that any enclosed block that is not filled, gets filled. See ARC prize puzzle 0062c1b

X (input)

To break down the problem for brute-force reductionism, the input X will be used in a loop against each y value. Lets say the grid translates to this list[list[int]] for X: 

        [
            [0, 1, 0],
            [1, 0, 1],
            [0, 1, 0],
        ]

where 1 represents green and 0 unfilled.

The accompanying y output values translate to:

        [
            [0, 1, 0],
            [1, 2, 1],
            [0, 1, 0],
        ]

where the 2 represents orange.

Both X and y are then flattened to 1-dimension like so:

        X = [0, 1, 0, 1, 0, 1, 0, 1, 0]
        y = [0, 1, 0, 1, 2, 1, 0, 1, 0]

X is then fed to a program generator that finds opcodes to calculate y[n]. Let's walk through this:

We now have a set of programs that can take input X and solve each y[n] individually. You can think of this programatically like this (Python): 

    # track programs by signature of X
    mem = Memory()

    X = (0, 1, 0, 1, 0, 1, 0, 1, 0)
    y = (0, 1, 0, 1, 2, 1, 0, 1, 0)

    programs: list[Program] = []
    for y_item in y:
        program = VirtualMachine(
            X=X, y=[y_item]
        ).bruteforce()
        programs.append(program)
    mem.insert(X=X, y=y, programs=programs)
    mem.save_to_disk("/path/to/memfile")

This introduced the concept of the Mem() object that tracks programs and their signatures. The idea behind Mem is that it can be used to lookup exact matches for a set of X input, or fuzzy close matches. Example: 

    X = (0, 1, 0, 1, 0, 1, 0, 1, 0)
    programs = mem.find(X)
    y_output = VirtualMachine(X=X).run(programs)
    print(y_output) 
    # -> (0, 1, 0, 1, 2, 1, 0, 1, 0)

Early Observations

I have been testing against the ARC-AGI dataset found here.

Notable observations:

  • The number of programs generated for the entire dataset averaged around ~130. Meaning, only 130 programs were needed to solve all of the problems. Note that these are individual programs which are grouped, used in different configurations (ordering and length) depending on the problem.
  • The average brute-force time per X -> y[n] is 0.6 seconds.
  • The entire dataset was brute-forced in under 12 minutes on single-core CPU only. Using multiprocessing can greatly reduce this time.
  • An early attempt of brute-forcing X against the entirety of y (not y[n]), as expected, took forever (never finished in a reasonable time or ever in some cases).
